# 前置条件
- `.ai/进度状态.json` 中的 `current_step` 必须为 `E5`。
- `E4` 已完成（有变更需求，或用户已确认无变更）。

## 目标
分析变更请求对现有系统的影响范围，包括涉及的模块、数据模型、API、前端页面等，并输出影响分析报告。

## Action
- 读取变更请求文件（`.ai/变更请求/current.md`）。
- 读取现有项目文档：`AGENTS.md`、`data-model.json`、`.ai/config/modules.yaml`、`.ai/config/plugins.yaml` 以及已生成的代码。
- 根据变更描述，识别受影响的实体：
  - 数据模型：新增/修改实体、字段、关系。
  - API：新增/修改接口、参数、返回值。
  - 前端页面：新增/修改页面、组件、路由。
  - 业务逻辑：新增/修改计算规则、工作流。
- 分析变更类型（微调、扩展、重构）：
  - 微调（Tweak）：小改动，不影响核心结构。
  - 扩展（Extension）：新增功能或字段，不影响现有功能。
  - 重构（Refactor）：修改核心逻辑或数据结构，可能影响多个模块。
- 输出影响分析报告（`.ai/reports/impact_analysis.md`），包含：
  - 变更摘要
  - 受影响模块列表（及风险等级）
  - 建议的修改策略（增量更新或全量重跑）

## Assert
- 影响分析报告已生成。
- 所有受影响模块已被识别。

## Artefact
- 影响分析报告（`.ai/reports/impact_analysis.md`）。

## Analysis
- 若变更类型为重构，提示用户可能需要重新生成多个模块。
- 评估回归风险，建议增加测试覆盖。

## Adjust
- 如果变更描述不清晰，返回 E4 请求用户补充。
- 若无影响（如用户要求删除未使用的文件），直接标记为无需操作。

## Advance
- 更新 `.ai/进度状态.json`：
  ```json
  {
    "current_step": "E6",
    "steps_completed": ["...", "E5"]
  }
  ```
- 提示用户：下一步应执行 E6（重新生成受影响模块）。

## 干预
如果影响范围过大（超过5个模块），建议用户人工评估。
