# 前置条件
- `.ai/进度状态.json` 中的 `current_step` 必须为 `E9.8`。
- `E9.7` 已完成（回归测试已编写）。

## 目标
运行项目的全量测试套件（包括单元测试、集成测试、端到端测试），确保修复没有破坏现有功能，并验证回归测试通过。

## Action
- 读取项目测试配置（如 `pytest.ini`、`jest.config.js`）。
- 执行全量测试命令（如 `pytest tests/`、`npm test`）。
- 收集测试结果：总用例数、通过数、失败数、错误数、跳过数。
- 重点关注：
  - 本次修复相关的模块及其依赖模块的测试。
  - 新添加的回归测试必须通过。
- 生成全量测试报告（`.ai/缺陷报告/当前缺陷/全量测试报告.md`），包含：
  - 测试环境信息（操作系统、依赖版本）。
  - 失败用例的详细信息（名称、错误堆栈、预期 vs 实际）。
  - 测试覆盖率变化（如果可能）。

## Assert
- 全量测试通过率不低于修复前（原则上应 100% 通过）。
- 回归测试用例全部通过。

## Artefact
- 全量测试报告。

## Analysis
- 若发现新的失败用例，分析是否由本次修复引入：
  - 若是，回退到 E9.6 重新修复。
  - 若不是，记录为已有缺陷，但不阻塞本次修复。
- 若测试耗时过长，记录时间用于优化。

## Adjust
- 如果全量测试失败且无法自动修复，将模块状态回退至“生成中”，并提示用户手动介入。
- 若环境问题导致测试失败（如数据库未启动），尝试自动修复环境后重试。

## Advance
- 更新 `.ai/进度状态.json`：
  ```json
  {
    "current_step": "E9.9",
    "steps_completed": ["...", "E9.8"]
  }
  ```
- 提示用户：下一步应执行 E9.9（验证修复结果）。

## 干预
若全量测试失败且原因不明，需用户协助分析。
