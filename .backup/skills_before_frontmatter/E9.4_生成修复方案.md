# 前置条件
- `.ai/进度状态.json` 中的 `current_step` 必须为 `E9.4`。
- `E9.3` 已完成（根因已定位）。

## 目标
基于根因分析，生成一个或多个可行的修复方案，并评估每个方案的优缺点、风险和实现复杂度。

## Action
- 读取根因分析报告（`.ai/缺陷报告/当前缺陷/根因分析.md`）。
- 根据缺陷类型和影响范围，设计修复方案：
  - 代码逻辑错误：修正条件判断、循环、数据类型转换等。
  - 边界条件未处理：添加空值检查、范围校验、默认值。
  - 性能问题：添加缓存、优化查询、减少循环嵌套。
  - 安全问题：参数化查询、输入验证、权限检查。
  - 配置错误：修正环境变量、依赖版本、部署参数。
- 为每个方案提供：
  - 修改的代码文件及具体变更（diff 格式）。
  - 预期效果（是否彻底解决缺陷、是否有副作用）。
  - 实现复杂度（低/中/高）。
  - 风险（可能影响其他功能）。
- 若存在多种方案，按推荐度排序。
- 将修复方案写入 `.ai/缺陷报告/当前缺陷/修复方案.md`。

## Assert
- 至少生成一个修复方案。
- 方案描述清晰，可执行。

## Artefact
- 修复方案文档（`.ai/缺陷报告/当前缺陷/修复方案.md`）。

## Analysis
- 检查修复方案是否与已有代码风格和架构一致。
- 评估是否需要修改数据模型或 API 接口（若需要，标记为高风险）。

## Adjust
- 如果根因分析不完整，返回 E9.3 补充分析。
- 若方案中涉及第三方库升级，检查兼容性。

## Advance
- 更新 `.ai/进度状态.json`：
  ```json
  {
    "current_step": "E9.5",
    "steps_completed": ["...", "E9.4"]
  }
  ```
- 提示用户：下一步应执行 E9.5（用户选择方案）。

## 干预
用户需选择一个修复方案（或确认自动选择推荐方案）。
