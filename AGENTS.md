# 核心契约（不可修改）

> 本文件是项目的唯一行为准则。所有 AI Agent 必须严格遵循，不得偏离。一旦锁定，任何修改需经过用户明确同意。

## 0. 核心理念与胶水编程规则（最高准则）

### 0.1 八大核心理念
1. **确定性优先**：通过数据边界、标准化接口、功能模板、沙箱测试，将开发转化为配置工作。AI 只做匹配与填充，不自由发挥。
2. **核心契约唯一**：本文件是唯一行为准则，所有步骤必须遵守。AI 每一步前重新读取。
3. **半自动人机协同**：AI 负责繁重计算，人仅做简单选择（勾选、确认、试用）。
4. **模块化即插即用**：每个功能独立生成，失败隔离，可局部替换。界面与功能解耦。
5. **渐进式披露**：按阶段、按步骤加载文档，每次只加载当前所需最小信息。
6. **自动化经验沉淀**：AI 自动记录成功/失败教训，去重合并，跨项目复用。
7. **数据驱动**：所有功能需求、界面展示、业务逻辑必须基于 `data-model.json` 定义的实体、字段、关系。AI 不得自行创造数据边界外的字段或实体。
8. **双向向内收敛**：以确定的输入页面（数据生产端）和展示页面（数据消费端）为两个端点，通过图搜索算法在数据边界内寻找最优的数据转换路径（包括聚合、过滤、连接等），避免从零设计数据流。

### 0.2 胶水编程规则
- **禁止重复实现**：任何已有成熟模块（如认证、ORM、图表库）的功能，AI 不得自行编写代码，必须通过依赖引入。
- **只做连接与配置**：AI 生成的代码仅包含：调用第三方模块的 API、数据格式转换、配置参数填充、模块间路由编排。
- **禁止复制代码**：不得将第三方模块的源码复制到项目中修改。
- **模块接口标准化**：所有模块必须通过明确的 API 或配置集成，不得直接操作其他模块的内部状态。
- **依赖显式声明**：所有外部模块必须在 `package.json` / `requirements.txt` 中声明，并锁定版本。

违反上述任何一条的代码，视为无效，必须重新生成。

### 0.3 阶段行为边界（强制）

每个原子步骤只能执行其类型定义范围内的操作，不得越界。违反者视为流程违规，模块状态回退至"生成中"，并记录一次"越界警告"。

| 步骤类型 | 包含步骤示例 | 允许操作 | 禁止操作 |
|----------|-------------|----------|----------|
| 需求类 | A1-A3, B0.1, E4 | 提问、记录用户回答、更新需求文档 | 生成代码、设计数据库表、选择插件 |
| 设计类 | B0.2, B6-B8, E5 | 设计API、数据模型、选择插件、输出技术方案 | 编写具体代码、修改需求文档 |
| 编码类 | B9, C2, E6, E9.6 | 生成代码、测试用例、修改配置文件 | 修改需求文档、变更技术方案 |
| 审查类 | A14, B13, C5, D12 | 分析产物、生成报告、提出建议 | 修改代码、自动应用修复 |
| 测试类 | C1, C3, C4, C7.5 | 安装依赖、运行测试、收集结果 | 修改代码或测试用例 |
| 部署类 | D1-D11 | 生成部署脚本、上传、执行远程命令 | 修改核心代码、需求文档 |
| 清理类 | G1-G5 | 删除文件、移动文件、归档经验 | 删除未标记的永久文件 |

越界处理：AI立即丢弃禁止操作的内容，记录日志，重新执行。连续2次越界，模块状态改为"需人工介入"。

AI在执行任何原子步骤前，必须根据步骤类型加载**最小必要文档集**，不得加载无关内容。

## 1. 对抗性审查规则（强制）

### 1.0 对抗性审查的本质
对抗性审查不是普通的代码评审或质量检查，而是**以"挑错、攻击、找漏洞"为目的**，基于事实真相，不讨好用户，不放过任何可疑点。审查者与执行者的KPI完全对立，以确保客观。

**执行方式**：对抗性审查必须由独立的**审查者Agent**执行。该Agent与执行者Agent完全隔离，不得访问执行者的生成日志、中间代码或提示词。审查者Agent的KPI与执行者Agent对立。

### 1.1 审查范围
所有项目产出物在进入下一阶段前，必须通过对应的对抗性审查：

| 阶段 | 产出物 | 审查维度 | 审查者角色 |
|------|--------|----------|------------|
| 需求分析 | 需求文档、用户故事 | 完整性、一致性、可测试性、无歧义 | 需求审查者 |
| 数据建模 | 数据模型、实体关系图 | 范式、索引设计、字段类型、关系正确性 | 数据架构审查者 |
| 架构设计 | 模块划分、接口定义、技术选型 | 可扩展性、依赖合理性、技术栈一致性 | 架构审查者 |
| 代码生成 | 模块代码、测试用例 | 功能完整性、逻辑、安全、性能、可维护性 | 代码审查者 |
| 部署配置 | Dockerfile、CI/CD脚本、Nginx配置 | 安全性、环境隔离、幂等性 | 运维审查者 |
| 运维文档 | 监控、日志、备份方案 | 可操作性、故障恢复 | 运维审查者 |

### 1.2 双角色KPI（各环节独立）
- **执行者KPI**：用户满意度、修复响应时间、首次对抗审查通过率、交付代码的线上缺陷率。
- **审查者KPI**：发现的严重问题数、阻止的不合格产出数、审查覆盖率。
- **对立关系**：两者奖励函数相反，形成博弈。

### 1.3 效率保障规则
- **轮次上限**：每个产出物最多审查3轮。
- **质量优先**：不设硬性时间预算。
- **智能退出**：连续两轮问题递减，第3轮自动降低严格度（仅检查Critical和Major，忽略Minor）。
- **用户终裁**：若3轮后仍未通过，模块状态改为"需人工介入"。
- **经验反馈**：每次审查耗时被记录，用于优化审查策略。

### 1.4 角色隔离
- 审查者不得访问执行者的生成日志、提示词、中间版本代码。
- 审查者只能基于最终产物（代码文件、配置文件、文档）进行审查。

### 1.5 对抗性审查标准（强制）

审查必须按优先级顺序执行：**功能完整性 → 逻辑正确性 → 安全性 → 性能 → 可维护性 → 依赖配置**。前一维度未通过，不得进入下一维度。

#### 1.5.0 功能完整性（最高优先级，Critical）
- [ ] 输入页面：是否完整实现了需求文档中定义的所有字段、校验规则、提交动作？
- [ ] 展示页面：是否完整展示了需求文档中定义的所有字段、聚合计算、筛选条件？
- [ ] 数据转换：是否按照需求中的计算逻辑正确实现？
- [ ] 边界匹配：所有输入字段是否都能在 `data-model.json` 中找到对应实体字段？
- [ ] 输出验证：执行端到端测试，实际输出是否与预期完全一致？

**判定**：任何一项不通过，模块状态回退至"生成中"，执行者必须修复功能缺陷。

#### 1.5.1 逻辑正确性
- [ ] 边界条件是否处理？（空值、零值、极值、空列表、null）
- [ ] 异常路径是否有明确处理？
- [ ] 条件分支是否覆盖所有可能？
- [ ] 循环是否有正确的终止条件？
- [ ] 数据转换是否有精度丢失风险？

#### 1.5.2 安全性
- [ ] 是否存在 SQL 注入风险？
- [ ] 是否存在 XSS 漏洞？
- [ ] 是否存在未授权访问？
- [ ] 敏感信息是否硬编码？
- [ ] 依赖的第三方模块是否有已知高危漏洞？

#### 1.5.3 性能
- [ ] 是否存在 N+1 查询？
- [ ] 数据库查询是否使用了索引？
- [ ] 是否存在内存泄漏？
- [ ] 是否有不必要的重复计算？
- [ ] 前端是否有大量 DOM 操作或未节流的事件？

#### 1.5.4 可维护性（胶水编程规则）
- [ ] 是否违反胶水编程规则？
- [ ] 是否有重复代码块？
- [ ] 函数/模块是否过长？
- [ ] 命名是否清晰？
- [ ] 关键逻辑是否有中文注释？

#### 1.5.5 依赖与配置
- [ ] 所有外部依赖是否显式声明并锁定版本？
- [ ] 是否有循环依赖？
- [ ] 配置文件是否与环境分离？
- [ ] Dockerfile 是否遵循最佳实践？
- [ ] CI/CD 脚本是否幂等？

#### 1.5.6 严重等级判定
- **Critical**：功能完整性缺失、安全漏洞、数据损坏。必须修复。
- **Major**：性能严重问题、逻辑错误（非功能缺失）、可维护性差。建议修复。
- **Minor**：代码风格、注释不足、非关键依赖。可忽略。

### 1.6 缺陷修复强制规则
- **复现优先**：任何修复请求，必须先完成复现步骤。AI不得在无法复现的情况下猜测修复。
- **影响分析**：修复前必须输出影响范围报告。
- **回归测试**：修复后必须编写至少一个回归测试用例，并确保通过。
- **全量测试**：修复后必须运行受影响模块及其依赖模块的全部测试套件。
- **用户确认**：修复完成后，必须提供详细的验证步骤，由用户确认。
- **连续失败处理**：连续2次修复尝试均导致新缺陷，模块状态改为"需人工介入"。

## 9. 模块生命周期管理（强制规则）

### 9.1 状态流转
每个功能模块的状态必须严格按照以下路径流转，不得跳过或逆转：
规划中 → 生成中 → 待对抗审查 → 待验收 → 已验收 → 已废弃
↑ ↓ ↓
└──────────┴────────────┘ (审查未通过或用户要求修改时回退)

```
- **规划中**：模块已拆分，需求已明确，尚未生成代码。
- **生成中**：AI 正在生成模块代码、测试用例、配置。
- **待对抗审查**：代码生成完毕，等待**对抗性审查**（由独立审查者执行）。审查未通过则回退到"生成中"。
- **待验收**：对抗审查通过，等待用户试用确认。用户验收包括业务符合度、用户体验、主观偏好等。
- **已验收**：用户确认满足需求，模块进入"完成"状态。
- **已废弃**：用户明确不需要该模块，或迭代中被替换。
- **需人工介入**（临时状态）：当模块经过3轮审查仍未通过，或出现无法自动解决的冲突时，进入此状态。用户可决定：接受当前质量、要求继续修改、或废弃模块。

### 9.2 关键规则
- 模块一旦进入 **"已验收"** 状态，AI 必须将其从后续步骤的加载列表中移除，并在进度状态中标记 `archived: true`。
- **迭代修改**：若后续迭代需要修改已验收模块，必须先将其状态改为 **"规划中"**，再重新走生成流程。
- **废弃处理**：已废弃模块的代码不删除，但标记为 `deprecated`。
- **需人工介入**：该状态下的模块，AI 不得自动处理，必须等待用户指令。

### 9.3 状态与步骤映射
| 状态 | 对应步骤 | 备注 |
|------|----------|------|
| 规划中 | B6 | 初始状态 |
| 生成中 | B9 | 代码生成 |
| 待对抗审查 | C5 | 代码对抗性审查 |
| 待验收 | C8 | 用户试用并选择 |
| 已验收 | C9 | 锁定选中demo |
| 已废弃 | 用户主动标记或迭代替换 | 在迭代步骤E6中处理 |
| 需人工介入 | 任何审查环节超过3轮 | 等待用户决策 |

## 10. 冲突解决与仲裁机制

### 10.1 需求文档与数据模型不一致
- 当 `docs/requirements/` 中的需求与 `data-model.json` 不一致时，AI 必须：
  1. 暂停当前步骤，生成差异报告。
  2. 提示用户选择：更新需求文档，或更新数据模型。
  3. 用户选择后，AI 自动执行对应更新，然后继续。

### 10.2 执行者与审查者分歧
- 若执行者与审查者对某个问题的严重等级判定存在争议，AI 自动记录双方理由，并提交给用户终裁（通过人工干预点）。
- 用户的选择会作为经验记录，供未来类似问题自动参考。

## 11. 约束与规则（固定）
- 架构：边界先行、容器分离、三层架构
- 代码规范：camelCase, 2空格缩进, 中文注释
- 禁止：直接操作 DOM、绕过 API 访问数据库
- 变更记录：每次变更后运行 `./scripts/cleanup.sh` 并更新 `CHANGELOG.md`
- **文档-代码一致性检查**：每次代码变更后，AI 必须检查并更新对应的需求文档和技术方案。若用户手动修改代码，应通过自然语言指令要求 AI 同步文档。

## 12. 清理规则（强制）

### 12.1 需要清理的文件类型
| 文件类型 | 示例 | 清理时机 | 处理方式 |
|----------|------|----------|----------|
| 沙箱临时文件 | 日志、缓存、临时数据库 | 每次测试完成后 | 立即删除 |
| 中间产物 | `data-model.json` 草稿、未采纳demo | 阶段A/C结束 | 删除草稿，保留最终版本 |
| 审查报告 | `_adversarial_report_*.md` | 模块验收后 | 直接删除 |
| 待整理经验 | `.ai/experience-pending/*.md` | 交付前 | 归档到全局经验库后删除本地副本 |
| 临时配置文件 | `.ai/config/*.tmp.yaml` | 每次配置更新后 | 立即删除 |
| 预览环境临时文件 | 临时容器、端口映射 | 用户选定demo后 | 自动销毁，清理临时配置 |

### 12.2 清理规则执行
- **自动清理**：AI 在每个阶段结束后自动执行。
- **直接删除**：不使用垃圾桶机制，所有临时文件直接删除。
- **容量上限**：若 `.ai/cache/` 目录超过 500MB，AI 自动清理最旧的缓存文件。
- **交付前强制清理（按顺序执行）**：
  1. 整理经验：将 `.ai/experience-pending/` 中的经验去重合并，归档到全局经验库。
  2. 归档后删除本地经验目录。
  3. 清理临时目录：删除 `.ai/tmp/`、`.ai/cache/`、`.ai/reports/`。
  4. 清理需求文档（可选）：删除 `docs/requirements/`，除非用户创建 `.keep_requirements` 文件。
  5. 清理其他临时文件：删除测试、预览环境残留配置。
  6. 确保最终交付物只包含源代码、README、可选 `AGENTS.md` 和 `data-model.json`。

### 12.3 禁止保留的文件
- 任何包含"草稿""tmp""backup""old"字样的文件或目录。
- 沙箱运行日志、测试覆盖率报告。
- 未使用的插件、依赖包源码。
- 用户手动创建的临时笔记（除非明确标记为"保留"）。

### 12.4 清理脚本
AI 在项目初始化时应生成 `./scripts/cleanup.sh`（Linux/macOS）和 `./scripts/cleanup.ps1`（Windows），用户也可手动执行。

### 12.5 异常处理
- 若清理过程中遇到文件占用无法删除，AI 应记录错误并跳过，在交付总结中提醒用户手动处理。

### 13.2 强制执行方式
- AI 在执行上述步骤前，必须检查是否已经调用过 `创建备份分支`（通过检查 `.ai/last_backup_branch.txt` 是否存在且时间戳在本次会话内）。
- 如果未备份，AI 应自动调用 `创建备份分支` 技能，并等待备份完成后再继续。
- 备份分支命名格式：`backup/YYYYMMDD-HHMMSS-<步骤名称>`（例如 `backup/20250403-153000-A9`）。

### 13.3 回滚提示
- 如果任何步骤执行后用户发现错误，AI 应提示："是否需要回滚到最近备份？请运行 `回滚到备份分支` 技能。"
- 回滚后，AI 应重新执行失败的步骤（需用户确认）。

### 13.4 用户豁免
- 高级用户可通过在步骤指令中添加 `--no-backup` 参数跳过备份（例如："请执行 A9_应用修正.md --no-backup"）。
- 跳过备份时，AI 必须输出警告："⚠️ 跳过备份，后续无法自动回滚，风险自担。"

## 14. 全局经验库使用规范

### 14.1 存储路径
- 默认路径：`~/.ai_experience/`
- 可通过环境变量 `AI_EXPERIENCE_PATH` 覆盖。
- 目录结构：`index.json`、`frontend/`、`backend/`、`database/`、`deployment/`、`general/`、`global-rules/`、`defect-patterns/`。

### 14.2 经验写入规则
- **写入时机**：项目交付前整理待整理经验；用户标记为"值得全局复用"；经验在3个以上项目中验证有效。
- **写入格式**：Markdown 文件，包含项目上下文、问题/现象、解决方案、标签、验证次数。
- **索引更新**：每次写入后自动更新 `index.json`。

### 14.3 经验查询规则
- **查询时机**：步骤B7、B9、C5、E5前。
- **查询方法**：根据技术栈、领域、步骤ID匹配，按验证次数降序取前5。
- **冲突处理**：矛盾经验同时展示，以当前项目 `AGENTS.md` 为准。

### 14.4 全局规则提升
- **提升条件**：验证次数≥3，无矛盾，可形式化。
- **提升方式**：用户确认后写入 `global-rules/common-constraints.yaml`，新项目自动引用。

### 14.5 跨项目复用
- 新项目初始化时加载 `index.json` 和 `global-rules/`，按需加载具体内容。

### 14.6 清理与维护
- 每季度清理超过2年未被引用的经验。
- 自动去重合并，冲突仲裁。

### 14.7 安全与隐私
- 全局经验库不得包含敏感信息，AI写入前自动脱敏。
- 文件权限建议 `700`。
