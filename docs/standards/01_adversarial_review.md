## 1. 对抗性审查规则（强制）

### 1.0 对抗性审查的本质
对抗性审查不是普通的代码评审或质量检查，而是**以"挑错、攻击、找漏洞"为目的**，基于事实真相，不讨好用户，不放过任何可疑点。审查者与执行者的KPI完全对立，以确保客观。

**执行方式**：对抗性审查必须由独立的**审查者Agent**执行。该Agent与执行者Agent完全隔离，不得访问执行者的生成日志、中间代码或提示词。审查者Agent的KPI与执行者Agent对立。

### 1.1 审查范围
所有项目产出物在进入下一阶段前，必须通过对应的对抗性审查：

| 阶段 | 产出物 | 审查维度 | 审查者角色 |
|------|--------|----------|------------|
| 需求分析 | 需求文档、用户故事 | 完整性、一致性、可测试性、无歧义 | 需求审查者 |
| 数据建模 | 数据模型、实体关系图 | 范式、索引设计、字段类型、关系正确性 | 数据架构审查者 |
| 架构设计 | 模块划分、接口定义、技术选型 | 可扩展性、依赖合理性、技术栈一致性 | 架构审查者 |
| 代码生成 | 模块代码、测试用例 | 功能完整性、逻辑、安全、性能、可维护性 | 代码审查者 |
| 部署配置 | Dockerfile、CI/CD脚本、Nginx配置 | 安全性、环境隔离、幂等性 | 运维审查者 |
| 运维文档 | 监控、日志、备份方案 | 可操作性、故障恢复 | 运维审查者 |

### 1.2 双角色KPI（各环节独立）
- **执行者KPI**：用户满意度、修复响应时间、首次对抗审查通过率、交付代码的线上缺陷率。
- **审查者KPI**：发现的严重问题数、阻止的不合格产出数、审查覆盖率。
- **对立关系**：两者奖励函数相反，形成博弈。

### 1.3 效率保障规则
- **轮次上限**：每个产出物最多审查3轮。
- **质量优先**：不设硬性时间预算。
- **智能退出**：连续两轮问题递减，第3轮自动降低严格度（仅检查Critical和Major，忽略Minor）。
- **用户终裁**：若3轮后仍未通过，模块状态改为"需人工介入"。
- **经验反馈**：每次审查耗时被记录，用于优化审查策略。

### 1.4 角色隔离
- 审查者不得访问执行者的生成日志、提示词、中间版本代码。
- 审查者只能基于最终产物（代码文件、配置文件、文档）进行审查。

### 1.5 对抗性审查标准（强制）

审查必须按优先级顺序执行：**功能完整性 → 逻辑正确性 → 安全性 → 性能 → 可维护性 → 依赖配置**。前一维度未通过，不得进入下一维度。

#### 1.5.0 功能完整性（最高优先级，Critical）
- [ ] 输入页面：是否完整实现了需求文档中定义的所有字段、校验规则、提交动作？
- [ ] 展示页面：是否完整展示了需求文档中定义的所有字段、聚合计算、筛选条件？
- [ ] 数据转换：是否按照需求中的计算逻辑正确实现？
- [ ] 边界匹配：所有输入字段是否都能在 `data-model.json` 中找到对应实体字段？
- [ ] 输出验证：执行端到端测试，实际输出是否与预期完全一致？

**判定**：任何一项不通过，模块状态回退至"生成中"，执行者必须修复功能缺陷。

#### 1.5.1 逻辑正确性
- [ ] 边界条件是否处理？（空值、零值、极值、空列表、null）
- [ ] 异常路径是否有明确处理？
- [ ] 条件分支是否覆盖所有可能？
- [ ] 循环是否有正确的终止条件？
- [ ] 数据转换是否有精度丢失风险？

#### 1.5.2 安全性
- [ ] 是否存在 SQL 注入风险？
- [ ] 是否存在 XSS 漏洞？
- [ ] 是否存在未授权访问？
- [ ] 敏感信息是否硬编码？
- [ ] 依赖的第三方模块是否有已知高危漏洞？

#### 1.5.3 性能
- [ ] 是否存在 N+1 查询？
- [ ] 数据库查询是否使用了索引？
- [ ] 是否存在内存泄漏？
- [ ] 是否有不必要的重复计算？
- [ ] 前端是否有大量 DOM 操作或未节流的事件？

#### 1.5.4 可维护性（胶水编程规则）
- [ ] 是否违反胶水编程规则？
- [ ] 是否有重复代码块？
- [ ] 函数/模块是否过长？
- [ ] 命名是否清晰？
- [ ] 关键逻辑是否有中文注释？

#### 1.5.5 依赖与配置
- [ ] 所有外部依赖是否显式声明并锁定版本？
- [ ] 是否有循环依赖？
- [ ] 配置文件是否与环境分离？
- [ ] Dockerfile 是否遵循最佳实践？
- [ ] CI/CD 脚本是否幂等？

#### 1.5.6 严重等级判定
- **Critical**：功能完整性缺失、安全漏洞、数据损坏。必须修复。
- **Major**：性能严重问题、逻辑错误（非功能缺失）、可维护性差。建议修复。
- **Minor**：代码风格、注释不足、非关键依赖。可忽略。

### 1.6 强制执行验证规则（防止AI忽略断言）

为确保关键断言不被AI忽略，以下步骤必须通过外部命令行工具验证，不得仅依赖AI自觉判断。

| 步骤 | 验证内容 | 验证命令 | 失败处理 |
|------|----------|----------|----------|
| A7（生成数据模型草稿） | JSON格式有效 | `cat .ai/tmp/data-model-draft.json \| jq empty` | 中止步骤，提示用户检查JSON语法 |
| A11（保存最终数据模型） | 符合JSON Schema | `check-jsonschema --schemafile shared/schemas/data-model.schema.json data-model.json` | 中止，显示违反的规则 |
| B9（生成模块代码） | Python代码规范（如适用） | `flake8 backend/ --max-line-length=120` | 中止，显示违规行 |
| B9（生成模块代码） | JS代码规范（如适用） | `eslint frontend/ --rule 'camelcase: error'` | 中止，显示违规规则 |
| C3（运行沙箱测试） | 所有测试通过 | `pytest tests/ --maxfail=1 -q` | 中止，显示失败用例 |
| C5（代码对抗性审查） | 无Critical问题（安全扫描） | `bandit -r backend/ -f json -o .ai/reports/bandit.json` | 生成报告，若存在HIGH或CRITICAL问题则中止 |
| G3（删除中间文件前） | 路径长度检查 | `find . -type f -path '*/\.*' -prune -o -type f -exec bash -c 'len=$(wc -c <<< "$1"); if [ $len -gt 260 ]; then echo "$1: $len"; fi' _ {} \;` | 警告用户，但不自动中止（由用户决定） |

**AI行为强制要求**：
- 执行上述步骤时，AI必须运行对应的验证命令，并检查退出码（0为成功，非0为失败）。
- 如果验证失败，AI必须输出错误信息，并中止当前步骤，不得继续执行 `## Advance` 或进入下一步。
- 如果验证工具未安装，AI必须提示用户安装命令（例如 `pip install check-jsonschema`），并中止步骤。

**豁免情况**：
- 用户明确使用 `--skip-validation` 参数时，可跳过验证（例如："请执行B9 --skip-validation"）。
- 跳过验证时，AI必须输出警告："⚠️ 跳过验证，请自行确保产物正确。"

### 1.7 缺陷修复强制规则
- **复现优先**：任何修复请求，必须先完成复现步骤。AI不得在无法复现的情况下猜测修复。
- **影响分析**：修复前必须输出影响范围报告。
- **回归测试**：修复后必须编写至少一个回归测试用例，并确保通过。
- **全量测试**：修复后必须运行受影响模块及其依赖模块的全部测试套件。
- **用户确认**：修复完成后，必须提供详细的验证步骤，由用户确认。
- **连续失败处理**：连续2次修复尝试均导致新缺陷，模块状态改为"需人工介入"。
