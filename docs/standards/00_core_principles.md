## 0. 核心理念与胶水编程规则（最高准则）

### 0.1 八大核心理念
1. **确定性优先**：通过数据边界、标准化接口、功能模板、沙箱测试，将开发转化为配置工作。AI 只做匹配与填充，不自由发挥。
2. **核心契约唯一**：本文件是唯一行为准则，所有步骤必须遵守。AI 每一步前重新读取。
3. **半自动人机协同**：AI 负责繁重计算，人仅做简单选择（勾选、确认、试用）。
4. **模块化即插即用**：每个功能独立生成，失败隔离，可局部替换。界面与功能解耦。
5. **渐进式披露**：按阶段、按步骤加载文档，每次只加载当前所需最小信息。
6. **自动化经验沉淀**：AI 自动记录成功/失败教训，去重合并，跨项目复用。
7. **数据驱动**：所有功能需求、界面展示、业务逻辑必须基于 `data-model.json` 定义的实体、字段、关系。AI 不得自行创造数据边界外的字段或实体。
8. **双向向内收敛**：以确定的输入页面（数据生产端）和展示页面（数据消费端）为两个端点，在数据边界内，通过AI的逻辑补全能力，自动填充中间层（接口层、功能层），避免从零设计数据流。
9. **四层分离架构**：数据层（数据库）、接口层（API）、功能层（业务逻辑）、界面层（前端）严格分离。AI不创造逻辑，而是作为"逻辑补全和配置工具"，将两端确定的需求转化为可执行的代码和配置。

### 0.2 胶水编程规则
- **禁止重复实现**：任何已有成熟模块（如认证、ORM、图表库）的功能，AI 不得自行编写代码，必须通过依赖引入。
- **只做连接与配置**：AI 生成的代码仅包含：调用第三方模块的 API、数据格式转换、配置参数填充、模块间路由编排。
- **禁止复制代码**：不得将第三方模块的源码复制到项目中修改。
- **模块接口标准化**：所有模块必须通过明确的 API 或配置集成，不得直接操作其他模块的内部状态。
- **依赖显式声明**：所有外部模块必须在 `package.json` / `requirements.txt` 中声明，并锁定版本。

违反上述任何一条的代码，视为无效，必须重新生成。

### 0.3 阶段行为边界（强制）

每个原子步骤只能执行其类型定义范围内的操作，不得越界。违反者视为流程违规，模块状态回退至"生成中"，并记录一次"越界警告"。

| 步骤类型 | 包含步骤示例 | 允许操作 | 禁止操作 |
|----------|-------------|----------|----------|
| 需求类 | A1-A3, B0.1, E4 | 提问、记录用户回答、更新需求文档 | 生成代码、设计数据库表、选择插件 |
| 设计类 | B0.2, B6-B8, E5 | 设计API、数据模型、选择插件、输出技术方案 | 编写具体代码、修改需求文档 |
| 编码类 | B9, C2, E6, E9.6 | 生成代码、测试用例、修改配置文件 | 修改需求文档、变更技术方案 |
| 审查类 | A14, B13, C5, D12 | 分析产物、生成报告、提出建议 | 修改代码、自动应用修复 |
| 测试类 | C1, C3, C4, C7.5 | 安装依赖、运行测试、收集结果 | 修改代码或测试用例 |
| 部署类 | D1-D11 | 生成部署脚本、上传、执行远程命令 | 修改核心代码、需求文档 |
| 清理类 | G1-G5 | 删除文件、移动文件、归档经验 | 删除未标记的永久文件 |

越界处理：AI立即丢弃禁止操作的内容，记录日志，重新执行。连续2次越界，模块状态改为"需人工介入"。

**双向收敛体现**：
- 需求类步骤（A1-A3）负责确定"两端"：输入页面（数据生产端）和展示页面（数据消费端）。
- 设计类步骤（B0.2, B6-B8）负责在数据边界内，从两端向中间收敛，自动推导接口层和功能层的实现。
- AI不创造新的数据结构或业务逻辑，仅根据两端定义和现有数据模型，补全中间转换路径。

### 0.4 上下文加载规则（按需渐进）

AI在执行任何原子步骤前，必须根据步骤类型加载**最小必要文档集**，不得加载无关内容。

| 步骤类型 | 必须加载 | 可选加载 | 禁止加载 |
|----------|----------|----------|----------|
| 需求类 | `AGENTS.md` 第0-1章 | 无 | `data-model.json`、其他模块spec |
| 设计类 | `AGENTS.md` 第0-3章、`data-model.json`、当前模块需求 | 相关模块API契约 | 其他模块spec |
| 编码类 | `AGENTS.md` 第0-7章、`data-model.json`、当前模块spec+技术方案 | 相关模块接口 | 其他模块spec |
| 审查类 | 当前产物 + `AGENTS.md` 第1章 | 被审查模块spec | 其他模块 |
| 测试类 | `AGENTS.md` 第0-1章、当前模块代码+测试 | 依赖模块stub | 其他模块spec |
| 部署类 | `AGENTS.md` 第0-1章、第8章、`.ai/config/deployment.yaml` | 无 | 所有模块spec |
| 清理类 | `AGENTS.md` 第12章 | 无 | 所有模块spec |

加载方式：优先使用状态缓存，分段加载，懒加载。容量监控：若上下文使用超过模型上限的80%，AI应暂停并提示用户开启新窗口。

### 0.5 主要矛盾识别原则
- 在任何决策点，AI必须首先识别当前阶段的主要矛盾（阻碍进展的关键问题）。
- 主要矛盾是指：如果该问题不解决，其他所有工作都无法推进或失去意义。
- 优先解决Critical级别问题，再处理Major和Minor。
- 用户可在A8步骤（展示数据边界摘要）后确认核心功能优先级，AI应据此调整后续生成顺序。

### 0.6 AI角色定位（非创造，是补全）
- AI不是创造者，而是**逻辑补全和配置工具**。
- 所有业务逻辑、数据结构、接口定义必须由用户或数据模型确定，AI只负责：
  - 将用户的自然语言需求转化为结构化配置。
  - 根据数据边界和页面需求，自动生成中间层代码（胶水代码）。
  - 选择合适的插件和模块进行组装。
- 禁止AI自行设计超出数据边界的字段、实体或业务规则。
