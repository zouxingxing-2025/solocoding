# 前置条件
- `.ai/进度状态.json` 中的 `current_step` 必须为 `D12`。
- `D11` 已完成。

## 目标
由独立的审查者Agent对部署配置文件（Dockerfile、CI/CD脚本、Nginx配置等）进行对抗性审查，检查安全性、环境隔离、幂等性等维度。

## Action
- 审查者Agent（独立上下文）加载：
  - `AGENTS.md` 第1章（审查标准）
  - 部署配置文件：`.ai/config/deployment.yaml`、`Dockerfile`、`docker-compose.yml`、CI/CD脚本（`.github/workflows/deploy.yml`等）、Nginx配置片段
- 按以下顺序检查：
  1. 安全性：是否暴露敏感端口？是否以 root 用户运行？是否包含硬编码密钥？
  2. 环境隔离：开发/生产配置是否分离？是否使用环境变量？
  3. 幂等性：重复执行部署脚本是否会产生副作用（如重复添加路由）？
  4. 可维护性：配置文件是否清晰？是否有注释？
  5. 依赖与版本：基础镜像是否固定标签（非 `latest`）？依赖版本是否锁定？
- 生成问题列表（含严重等级、位置、建议修复方案）。
- 输出审查报告到 `.ai/reports/deploy_review.md`。

## Assert
- 审查报告已生成，包含所有检查维度。
- 若没有发现问题，报告仅包含"通过"。

## Artefact
- 审查报告（`.ai/reports/deploy_review.md`）。

## Analysis
- 匹配缺陷模式库中的部署相关模式（如"Dockerfile 使用 root 用户"）。
- 记录常见问题，用于优化部署模板。

## Adjust
- 若存在 Critical 问题，将部署配置状态回退至 D1，要求执行者修复。
- 若只有 Major/Minor，生成修复建议，由执行者决定是否修复（可通过用户终裁）。

## Advance
- 更新 `.ai/进度状态.json`：
  ```json
  {
    "current_step": "E1",
    "steps_completed": ["...", "D12"]
  }
  ```
- 提示用户：阶段4完成，下一步进入阶段5（运维与迭代），应执行 E1。

## 干预
如果审查者与执行者对问题严重等级有争议，提交用户终裁。
