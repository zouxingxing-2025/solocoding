# 代码对抗性审查

## 前置条件
- `.ai/进度状态.json` 中的 `current_step` 必须为 `E5`。
- `E4` 已完成（测试通过或重试成功）。

## 目标
由独立的审查者Agent对生成的代码进行对抗性审查，从功能完整性、业务规则实现、逻辑正确性、安全性、性能、可维护性、依赖配置等维度挑错。

## Action
- 审查者Agent（独立上下文）加载：
  - `AGENTS.md` 第1章（审查标准）
  - 当前模块的代码文件、测试文件、配置文件
  - 模块的需求文档和技术方案
  - 业务规则文件（`.ai/temp/business_rules.json`）
- 按以下顺序检查：
  1. **功能完整性**：是否实现了所有验收标准和业务规则？字段、API、交互是否完整？
  2. **业务规则实现**：是否正确实现了数据校验、流程约束、权限控制？
  3. **逻辑正确性**：边界条件、异常处理、分支覆盖？
  4. **安全性**：SQL注入、XSS、未授权访问、硬编码密钥？
  5. **性能**：N+1查询、索引使用、内存泄漏？
  6. **可维护性**：符合胶水编程规则？代码重复？命名清晰？
  7. **依赖与配置**：依赖声明完整？版本锁定？配置分离？
- 生成问题列表（含严重等级、位置、建议修复方案）。
- 输出审查报告到 `.ai/reports/code_review.md`。

## Assert
- 审查报告已生成，包含所有检查维度。
- 若没有发现问题，报告仅包含"通过"。

## Artefact
- 审查报告（`.ai/reports/code_review.md`）。

## Analysis
- 匹配缺陷模式库（`~/.ai_experience/defect-patterns/`），推荐修复方案。
- 记录常见问题模式，用于后续优化。

## Adjust
- 若存在 Critical 问题（如业务规则未实现、安全漏洞），将模块状态回退至"生成中"，要求执行者修复。
- 若只有 Major/Minor，生成修复建议，由执行者决定是否修复（可通过用户终裁）。

## Advance
- 更新 .ai/进度状态.json：
  ```json
  {
    "current_step": "E6",
    "steps_completed": ["...", "E5"]
  }
  ```
- 提示用户：下一步应执行 E6。

## 干预
- 如果审查者与执行者对问题严重等级有争议，提交用户终裁。
