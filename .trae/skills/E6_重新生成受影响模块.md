# 前置条件
- `.ai/进度状态.json` 中的 `current_step` 必须为 `E6`。
- `E5` 已完成（影响范围分析已输出）。

## 目标
根据影响分析报告，重新生成受变更影响的模块（包括代码、测试、配置），保持与现有系统的兼容性。

## Action
- 读取 `.ai/reports/impact_analysis.md`，获取受影响模块列表。
- 对于每个受影响的模块：
  - 若变更类型为"微调"或"扩展"，仅重新生成修改部分（如增量代码、新增字段）。
  - 若为"重构"，则全量重新生成该模块（调用 B9 生成模块代码）。
- 合并新生成的代码与现有代码：
  - 保留未受影响的部分。
  - 更新模块的版本号和变更日志。
- 更新 `.ai/config/modules.yaml` 中对应模块的状态为"生成中"。
- 运行增量测试（仅测试受影响模块及其依赖模块），确保变更未破坏现有功能。
- 记录重新生成日志（`.ai/logs/regenerate.log`）。

## Assert
- 所有受影响模块已重新生成。
- 增量测试通过。

## Artefact
- 更新后的模块代码。
- 增量测试报告（`.ai/reports/regression_test.md`）。

## Analysis
- 检查新生成的代码是否与现有接口兼容（如 API 签名、数据模型）。
- 若发现冲突，自动解决或提示用户。

## Adjust
- 如果重新生成失败，回退到 B0.1 重新进行功能需求澄清。
- 若增量测试失败，定位问题并再次修复。

## Advance
- 更新 `.ai/进度状态.json`：
  ```json
  {
    "current_step": "E7",
    "steps_completed": ["...", "E6"]
  }
  ```
- 提示用户：下一步应执行 E7（执行蓝绿部署或滚动更新）。

## 干预
如果模块重新生成导致大量冲突，需用户介入决策。
