# 前置条件
- `.ai/进度状态.json` 中的 `current_step` 必须为 `E9.7`。
- `E9.6` 已完成（修复已执行）。

## 目标
为本次缺陷修复编写回归测试用例，确保缺陷不会再次出现，并验证修复未破坏现有功能。

## Action
- 读取缺陷描述、复现步骤和修复方案。
- 根据缺陷触发条件，生成至少一个回归测试用例：
  - 单元测试：针对修复的函数或模块，测试正常路径和边界条件。
  - 集成测试：模拟用户操作或 API 调用，验证完整流程。
- 将测试用例添加到项目的测试套件中（如 `tests/regression/test_<defect_id>.py`）。
- 运行新添加的回归测试，确保失败（红色阶段），然后运行修复后的代码，确保通过（绿色阶段）。
- 若测试框架支持，将测试用例标记为 `@regression` 以便后续区分。
- 记录测试用例的代码和运行结果到 `.ai/缺陷报告/当前缺陷/回归测试.md`。

## Assert
- 回归测试用例能够捕获原始缺陷（修复前失败，修复后通过）。
- 测试用例已纳入版本控制。

## Artefact
- 回归测试代码文件。
- 回归测试报告（`.ai/缺陷报告/当前缺陷/回归测试.md`）。

## Analysis
- 检查测试用例是否覆盖了类似的潜在缺陷（如其他边界条件）。
- 若测试用例编写复杂度过高，提示用户可能需要手动测试。

## Adjust
- 如果回归测试无法自动化（如 UI 交互），生成手动测试步骤清单。
- 若运行失败，检查修复是否完整，必要时回退到 E9.6。

## Advance
- 更新 `.ai/进度状态.json`：
  ```json
  {
    "current_step": "E9.8",
    "steps_completed": ["...", "E9.7"]
  }
  ```
- 提示用户：下一步应执行 E9.8（运行全量测试）。

## 干预
无。
