# 前置条件
- `.ai/进度状态.json` 中的 `current_step` 必须为 `E3`。
- `E2` 已完成（日志收集已配置）。

## 目标
为应用设置监控和告警机制，包括系统资源监控、应用性能监控、错误率告警等，确保能及时发现并响应问题。

## Action
- 根据 `AGENTS.md` 第2章技术栈和项目规模，选择合适的监控方案：
  - 轻量级：使用 `Prometheus` + `Grafana` 或云服务（如阿里云监控、AWS CloudWatch）。
  - 简单场景：配置 `healthcheck` 脚本 + 定时任务 + 邮件/钉钉通知。
- 生成监控配置文件（如 `prometheus.yml`，定义抓取指标的目标和频率）。
- 生成告警规则文件（如 `alerts.yml`），定义告警条件（如 CPU > 80%、5xx 错误率 > 1%、API 响应时间 > 2s）。
- 配置告警通知渠道（邮件、Webhook、钉钉、Slack 等），将接收地址写入 `.env` 或部署配置。
- 将配置文件上传到服务器相应目录（如 `/etc/prometheus/`），并重启监控服务。
- 提供监控面板的访问地址和默认登录凭证（首次登录需修改密码）。

## Assert
- 监控服务正常运行，能采集到指标数据。
- 测试告警条件（如模拟高负载）能触发通知。

## Artefact
- 监控配置文件（`.ai/config/monitoring/` 下）
- 告警规则文件
- 监控部署日志（`.ai/logs/monitoring_setup.log`）

## Analysis
- 检查监控指标是否覆盖关键维度（CPU、内存、磁盘、网络、应用 QPS、错误率、延迟）。
- 验证告警阈值是否合理，避免过于敏感或迟钝。

## Adjust
- 如果服务器资源有限，可选择使用轻量级方案（如 `node_exporter` + `cron` + 邮件）。
- 若用户无固定 IP 无法接收告警，建议使用钉钉/微信机器人。

## Advance
- 更新 `.ai/进度状态.json`：
  ```json
  {
    "current_step": "E4",
    "steps_completed": ["...", "E3"]
  }
  ```
- 提示用户：下一步应执行 E4（用户提交变更需求）。

## 干预
用户需提供告警接收地址（如邮箱、钉钉 webhook）。

若用户选择跳过监控，可标记 monitoring_enabled: false。


## When to Use This Skill
Use this skill when E3_设置监控告警 is required, such as during phase E tasks.

## Not For
- This skill does not handle other phases or unrelated tasks.
