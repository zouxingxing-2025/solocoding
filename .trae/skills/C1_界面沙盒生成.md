# 界面沙盒生成

## 前置条件
- `.ai/进度状态.json` 中的 `current_step` 为 `C1`。
- 数据模型（`data-model.json`）、业务规则（`.ai/temp/business_rules.json`）、角色任务数据（`.ai/temp/roles_tasks_fields.json`）已确认。
- 数学美学中间层已生成设计令牌（`docs/design-tokens.css`）。

## 目标
生成可交互的HTML界面原型（包含列表页、表单页、详情页），应用数学美学设计令牌（间距、圆角、配色、焦点），并集成业务规则校验。用户通过点击、填写等操作预览效果，确认后进入代码生成阶段。

## Action

### 1. 加载设计令牌
- 读取 `docs/design-tokens.css`，获取所有CSS变量（间距、字号、圆角、阴影、动画、配色比例等）。
- 确保生成的页面引用该CSS文件。

### 2. 根据角色生成多端界面
- 从 `.ai/temp/roles_tasks_fields.json` 中提取角色列表（如HR、候选人、经理）。
- 为每个角色生成独立的界面：
  - **HR端（后台管理）**：岗位管理、候选人列表、审批记录查看。
  - **候选人端（前端APP）**：岗位浏览、投递表单、我的投递。
  - **经理端（后台管理）**：待审批列表、审批表单。
- 使用相同的设计令牌，确保视觉统一。

### 3. 生成页面结构（HTML + CSS + JS）
- 使用 Flexbox/Grid 布局，宽度比例遵循黄金分割（如侧边栏38.2%，主内容61.8%）。
- 高频操作按钮（如"发布岗位"、"审批"）置于右下角黄金螺旋焦点。
- 高频字段（如岗位名称、城市）置于F型布局左上角。
- 低价值字段默认折叠（如"职位描述"），用户点击展开。
- 表单集成业务规则校验（实时提示，如"手机号格式错误"）。

### 4. 生成示例数据
- 根据数据模型，生成模拟数据（如岗位列表包含"Java工程师"、"产品经理"等）。
- 用于填充表格、表单下拉选项等。

### 5. 输出可交互原型
- 生成一个独立的HTML文件（`.ai/previews/app_preview.html`），包含所有角色的页面（通过标签页切换）。
- 用户点击角色标签，切换不同视图。
- 提供简单的模拟交互（如点击"投递"按钮弹出"提交成功"提示，不调用真实后端）。

### 6. 用户确认
- 在TRAE预览窗口中打开HTML文件。
- 向用户提问：
  ```markdown
  请预览以上界面，确认布局、操作流程和规则是否符合预期。
  - 如果满意，请回复"确认"。
  - 如果需要调整，请描述具体问题（如"岗位列表应该按城市分组"）。
  ```
等待用户反馈。如果需要修改，AI调整HTML/CSS/JS并重新生成预览（循环）。

用户确认后，保存最终原型作为后续代码生成的参考。

## Assert
- 所有角色对应的页面都已生成。
- 界面元素使用CSS变量，无硬编码样式。
- 高频操作按钮位于右下角，高频字段位于左上角。
- 业务规则校验已集成（如手机号格式提示）。

## Artefact
- .ai/previews/app_preview.html
- .ai/previews/design-tokens.css（已存在）

## Analysis
- 如果用户对布局不满意，AI可根据描述微调（如"将筛选栏移到顶部"）。
- 如果用户觉得操作复杂，AI可建议增加快捷操作（如批量审批）。

## Adjust
- 如果用户无法描述修改，AI可提供"布局模板"供选择（如"紧凑型"、"宽松型"）。
- 如果生成的原型文件过大，可拆分多个HTML（按角色），但统一入口。

## Advance
- 更新 .ai/进度状态.json，current_step = "D1"。
- 提示用户：下一步将生成生产代码。

## 干预
- 用户必须确认界面或提出修改。
