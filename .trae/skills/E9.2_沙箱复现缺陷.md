# E9.2_沙箱复现缺陷.md

## 前置条件
- `.ai/进度状态.json` 中的 `current_step` 必须为 `E9.2`。
- `E9.1` 已完成（已收集到复现步骤）。

## 目标
在沙箱环境中模拟按照用户提供的复现步骤执行操作，确认缺陷是否存在。若成功复现，记录复现日志；若无法复现，返回 E9.1 要求用户补充信息。

## Action
- 读取 `.ai/缺陷报告/当前缺陷/复现步骤.md` 中的操作步骤。
- 启动沙箱环境（与项目相同的运行时、数据库、配置）。
- 按步骤执行操作（模拟用户行为或调用 API）：
  - 若步骤包含前端交互，使用自动化工具（如 Playwright）或手动模拟。
  - 若步骤包含 API 调用，使用 `curl` 或 HTTP 客户端。
- 记录实际输出（操作内容、页面行为、错误日志）。
- 对比实际输出与用户描述的"实际结果"是否一致。
- 如果一致，则确认缺陷复现成功；否则标记为"无法复现"。

## Assert
- 沙箱环境已就绪，能模拟生产环境。
- 复现过程完整，无逻辑步骤。

## Artefact
- 复现日志（`.ai/缺陷报告/当前缺陷/复现日志.log`）。
- 复现结论（成功/失败）。

## Analysis
- 如果复现失败，分析原因：环境差异、步骤不清、用户描述错误。
- 若因环境差异导致失败，调整沙箱配置（如数据库版本、依赖）后重试。

## Adjust
- 若第一次复现失败，尝试调整步骤（如明确 URL 路径、参数）后重试，最多 3 次。
- 若仍无法复现，将结论写入报告，并返回 E9.1 请求用户补充信息。

## Advance
- 若复现成功，更新进度状态：
  ```json
  {
    "current_step": "E9.3",
    "steps_completed": ["...", "E9.2"]
  }
  ```
  提示用户：下一步应执行 E9.3（定位根因）。
- 若无法复现，将 current_step 回滚到 E9.1 并提示用户补充信息。

## 干预
无（自动执行，但若多次失败需用户输入）。
