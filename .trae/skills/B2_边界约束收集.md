# 边界约束收集

## 前置条件
- `.ai/进度状态.json` 中的 `current_step` 为 `B2`。
- 数据模型已确认（`data-model.json` 已生成）。

## 目标
根据数据模型和用户描述，自动推断候选业务规则（如字段格式、唯一性、流程约束），用户通过勾选框确认或修改。确保生成的APP符合业务逻辑，减少后期返工。

## Action

### 1. 读取数据模型和用户描述
- 读取 `data-model.json`，获取实体、字段、类型、关系。
- 读取用户工作流程描述（`.ai/进度状态.json` 中的 `workflow_desc`）和已确认的角色任务数据（`.ai/temp/roles_tasks_fields.json`）。

### 2. AI 推断候选规则
- 根据字段名、类型、业务语义，推断常见规则：
  - **格式校验**：手机号（11位数字）、邮箱（正则）、日期格式等。
  - **范围约束**：数值字段（如缺口数 ≥ 0）、金额（≥ 0）等。
  - **唯一性约束**：如用户名、手机号不能重复。
  - **流程约束**：如"岗位关闭后不能再投递"、"审批必须按顺序"。
  - **关联约束**：如"候选人不能重复投递同一岗位"。
- 每条规则附带**置信度**（高/中/低）和推断依据（自然语言）。

### 3. 生成规则确认界面
- 以勾选框列表形式展示候选规则，用户勾选启用、取消勾选禁用，或点击"修改"自定义。

**示例输出**：

```markdown
根据您的业务模型，我推测可能存在以下业务规则。请确认哪些需要启用：

✅ 岗位缺口数不能为负数（推断依据：字段"缺口数"通常应为正整数）【高置信度】
✅ 候选人不能重复投递同一岗位（推断依据：常见招聘系统去重规则）【中置信度】
❓ 岗位关闭后不能再投递（推断依据：字段"状态"可能包含"已关闭"，但不确定是否需要此规则，请确认）【低置信度】
✅ 手机号格式校验（推断依据：字段"手机号"通常需要11位数字格式）【高置信度】

如需修改规则，请点击"修改"并输入新规则。
```

### 4. 用户确认
等待用户回复。用户可：
- 直接回复"确认"，接受当前所有勾选状态。
- 回复"修改"，并描述需要调整的规则（如"将手机号格式改为允许座机"）。

如果用户提出修改，AI 更新规则列表，重新展示。

循环直到用户明确确认。

### 5. 保存规则
将最终确认的规则保存为 .ai/temp/business_rules.json，格式：

```json
{
  "rules": [
    { "id": "RULE_001", "name": "岗位缺口数不能为负数", "type": "range", "severity": "error", "expression": "gap_count >= 0" },
    { "id": "RULE_002", "name": "候选人不能重复投递同一岗位", "type": "unique", "severity": "error", "expression": "unique(candidate_id, position_id)" }
  ]
}
```

## Assert
- 至少有一条规则被启用（如果没有，提示用户"未选择任何规则，可能影响数据准确性"）。
- 规则格式正确，可被后续代码生成器解析。

## Artefact
- .ai/temp/business_rules.json

## Analysis
- 如果 AI 推断的规则置信度过低（如 < 30%），可主动提问："是否需要添加其他业务规则？例如：..."
- 如果用户多次修改规则，记录修改模式，用于优化后续项目的推断（经验沉淀）。

## Adjust
- 如果用户无法描述自定义规则，提供常见规则模板（如"不能重复提交"、"金额不能为负"）。
- 如果用户对规则理解有困难，可提供"模拟场景"示例（如"如果岗位缺口数填-1，系统会提示错误"）。

## Advance
- 更新 .ai/进度状态.json，current_step = "C1"。
- 提示用户：下一步将生成界面预览。

## 干预
- 用户必须确认规则或提出修改。

---

**说明**：
- 本技能文件完全基于 AI 推断 + 用户确认，用户只需勾选或简单修改，无需手动编写规则表达式。
- 规则存储为结构化 JSON，供后续代码生成使用（如前端表单校验、后端逻辑）。
